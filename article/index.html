Procedural Tokens — encoding design decisions

Most design tokens store an answer. The interesting ones store the rule that produces the answer. This is a short walk from the first kind to the second: from raw values, to references that assign them a role, to procedural tokens that keep the reasoning attached and can re-run it when the inputs change.

1. A token is a name for a value

#0066cc is a hex code. It tells you a colour exists. It tells you nothing about why it's there.

A design token gives that value a name. CSS, components, docs — anything that needs the colour — read the name, not the hex. Change the value once and every reader updates.

That's the whole idea. But picking the right name is where design tokens get interesting: different names belong on different layers, and the layer they live on changes what they're allowed to mean. The rest of this article walks through those layers.

2. Value tokens — the atoms

A value token is the simplest kind: a name and a literal value. Crucially, the name describes what the value is, not what it means: values.gray800, values.blue500, type.base. There's no role attached yet — just raw material.

  • values.gray800 =
  • values.gray50 =
  • values.blue500 =
  • type.base =
  • radius.base =

A subtle but important rule: a value token never gets a semantic name. Calling one values.brand would smuggle a decision ("this is the brand colour") into a layer that's supposed to hold pure material. Edit values.gray800 to red and the name is still values.gray800 — a slight lie, but a localised one. Meaning belongs upstairs.

3. Reference tokens — give names meaning

A reference token points to a value token. Its job is to introduce meaning. Where values.gray800 only says "this hex," color.brand says "this is what we call our brand." Same colour, but the name now has a job in the system.

  • color.brand = — points at values.gray800
  • color.surface = — points at values.gray50
  • color.onSurface = — paired with color.surface; the text colour that goes on it
  • color.text = — points at color.onSurface (references can chain)

Notice what just happened. The value layer holds no opinions — values.gray800 is just a hex with a descriptive label. The colour layer is where the design system makes decisions: this grey is our brand, that surface and its paired text always travel together, links use color.interaction. Same mechanism each time — a ref. Click any ref chip and you'll see the candidates it could point at instead.

That's the stack so far: value tokens describe material, reference tokens assign role. But a reference still only tells you the choice. Not why it was made.

4. The problem with decisions

The weakness shows up the moment a token contains a decision instead of a fact. A fact is easy: values.gray800 = '#202126'. A decision is harder: color.buttonText = ref('values.white'). White compared to what? Chosen for what reason? Valid under what conditions?

Reference tokens capture the outcome of a decision but hide the reason. Once the token ships, that reasoning lives somewhere else — in a Figma comment, a PR description, a Slack thread — and then decays. Six months later nobody is sure why the choice was made, so nobody wants to touch it. The system remembers the answer and forgets the question.

That is the point where I want a third layer. Not a token that stores the chosen value, but a token that stores the decision procedure. In other words: make the question part of the system too. Then two useful things happen:

  • The decision can be re-run automatically when its inputs change.
  • Anyone reading the token — including future me — can see how the choice was made, without hunting through documentation.

The button label in the example panel already works that way. The brittle version would be color.buttonText = ref('values.white') — fine for as long as the button stays dark, wrong the moment it doesn't. Instead, it's bestContrastWith(color.brand, ramp): the rule re-runs against whatever the button background ends up being and picks the most readable step from the ramp. The token does not just name the answer. It exposes the reasoning that produced it.

Re-point the brand colour and watch the label re-decide:

color.brand =

Try pointing it at a light palette colour, then a dark one. The label flips with it — because the rule re-runs against the new background and picks the ramp step with the highest contrast. No documentation to chase down: the token is the documentation.

5. Procedural tokens — rules, not values

A procedural token is that third layer. Value tokens describe material. Reference tokens assign role. Procedural tokens encode reasoning. Read the rule and you can see the decision. Change the inputs and the system can make the decision again.

In this demo, that shows up in four different kinds of design choices: threshold, scale, palette, and state.

Contrast with a threshold

Start with the button border, because you can see the effect immediately. Its hairline outline is computed:

color.line = minContrastWith(color.surface, ramp, { ratio:  })

Where bestContrastWith picks the highest contrast available, minContrastWith picks the lowest step that still clears the threshold. The result is a soft line that's visible against the surface but never shouts. Drop the threshold and it fades; raise it and it darkens. In this demo the control snaps to the contrast levels the current ramp can actually produce, because the function is choosing from a discrete set of candidates. Flip the surface dark and it auto-inverts. The decision encoded in the token isn't "use this colour" — it's "use the faintest colour that's still readable as a boundary."

Spacing as a scale

Every gap inside the example panel — title to paragraph, paragraph to chart, chart to button — is a multiple of . Pull the base, the whole rhythm scales together via spacingScale. The rule already knew what to do.

A ramp between two intents

The seven bars in the chart aren't seven hand-picked colours. They're seven steps of a procedurally-generated ramp built with colorMix(color.surface, color.interaction) at increasing ratios. Move and the chart re-tones; re-point to a different palette colour and it re-tones again. The ramp is downstream of two decisions: which surface we sit on, and which colour we use to act.

Darker on hover

The hover state is a smaller example, but it shows the same idea. Hover the link in the example panel and you'll see a shade of color.interaction. That shade isn't stored — it's computed:

color.linkHover = darken(color.interaction, )

Move the amount. The hover state updates wherever it's used — no CSS edit, no rebuild.

6. So what

Once your tokens can hold rules, your design system gets safer to change. You stop hard-coding yesterday's answers and start preserving the logic that produced them. Change the brand colour, the surface, the base spacing, the contrast threshold — the system can recompute the dependent choices instead of asking a human to remember them.

That is the narrative shift I care about. Value tokens answer "what is it?" Reference tokens answer "what is it for?" Procedural tokens answer "why this, under these conditions?" When the why lives in the token, the system stays legible longer, and it keeps making sense even after the inputs move.
